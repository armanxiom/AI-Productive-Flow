Build Ideaso – An Offline-First, AI-Native Productivity App

## Product Overview
Ideaso is a mobile-first, offline-first productivity app for capturing ideas, organizing knowledge, managing tasks, and building workflows. It must be dramatically simpler than Notion, faster on mobile, and fully reliable offline. All user data lives locally first, syncs when online, and is never lost. AI turns rough thoughts into structured content (summaries, tasks, outlines) – not just a chatbot.

## Core Product Principles
- Offline-first by default (IndexedDB via Dexie, PWA).
- Mobile-first and fast (<2s interactions on average device).
- Simplicity over infinite flexibility – obvious UI with clear purpose per screen.
- AI that creates structure, not just conversation.
- Strong data safety (local encryption stubs, exports, version history).
- Beautiful but minimal dark UI (Tailwind, shadcn/ui, Framer Motion).
- Understandable in under 1 minute – no blank-canvas overwhelm.
- Search must be instant (local index).
- Reliable sync with conflict handling (last-writer-wins + revision history).

## Target Users
Students, content creators, freelancers, solo founders, small agencies, knowledge workers, and teams seeking a simpler Notion alternative.

## Architecture Rules
- **All data lives locally first.** Every mutation goes to Dexie, then optionally queues a sync item.
- **Offline writes never blocked.** Sync queue processes when online.
- **API routes** for sync (`/api/sync/push`, `/api/sync/pull`) and AI (`/api/ai`).
- **No localStorage for app data** – only IndexedDB.
- Code must be modular: components, lib, store folders.
- Prefer real implementation over pseudocode. All files paste‑ready.
- Include exact file names in the folder tree.

Implementation Instructions
1. Project Setup
Initialize Next.js with TypeScript, install all dependencies (see package.json above). Configure Tailwind with the custom dark palette. Copy globals.css and tailwind.config.ts as provided.

2. Core Libraries
Set up Dexie database with all tables in lib/db.ts. Export a singleton db.

Create Zustand store with fields: isOnline, sidebarOpen, activeWorkspaceId, searchQuery, lastSyncAt.

Implement lib/utils.ts with cn, generateId, formatDate, truncate.

3. Local-First Data Operations
Every page/component reads from Dexie (using db.notes.toArray(), etc.) and writes to Dexie. After a local write, call enqueueChange(action, table, recordId, payload) from lib/sync.ts. This automatically pushes if online.

4. Sync Engine
lib/sync.ts must:

Queue actions into syncQueue table.

pushQueuedItems() sends the queue to /api/sync/push as POST.

pullRemoteChanges(workspaceId) GETs /api/sync/pull, then merges using last-writer-wins (compare updatedAt).

Expose syncNow() that does push then pull, used by background interval (set every 30s) and manual sync button.

Server routes: /api/sync/push receives array of queue items and upserts into Supabase tables; /api/sync/pull returns all rows for that workspace since last sync (or full dataset), handling timestamps.

5. AI Features
lib/ai.ts exports functions that call /api/ai route with a prompt. The API route uses OpenAI (or compatible) to:

Summarize note

Clean up note (grammar, structure)

Convert note to tasks (list of tasks)

Turn rough idea into an outline
All AI results are stored in aiActionLog table.
